# 샘플 PDF 수집 기준 및 예외 케이스

## 1. 목적

HighlightNote MVP는 "사용자가 직접 표시한 하이라이트"를 기준으로 노트를 생성한다.  
따라서 샘플 PDF 수집 단계에서 가장 먼저 검증해야 하는 것은 문서 요약 품질이 아니라,
"하이라이트 annotation을 안정적으로 추출하고 텍스트와 연결할 수 있는가"이다.

이 문서는 샘플 PDF를 어떤 기준으로 수집하고 분류할지, 그리고 어떤 예외 케이스를 먼저 확보해야 하는지 정의한다.

## 1-1. 현재 저장소 상태

- 현재 저장소에는 실제 샘플 PDF 파일이 없다.
- 따라서 아래 인벤토리 표는 기준 템플릿이며, 실제 파일 확보 후 값으로 채워야 한다.
- 구현 착수 전 최소 `N1`, `N2`, `E1`, `E2`에 해당하는 샘플을 먼저 확보한다.

## 2. 수집 목표

- 최소 5개, 권장 8~10개의 샘플 PDF를 확보한다.
- 정상 케이스와 실패 케이스를 모두 포함한다.
- 한 종류의 문서에 치우치지 않도록 교재, 강의자료, 논문을 섞는다.
- 각 샘플마다 "하이라이트 존재 여부", "텍스트 레이어 존재 여부", "예상 난이도"를 기록한다.

## 3. 필수 수집 기준

### 문서 유형
- 전공 교재 PDF
- 강의 슬라이드 PDF
- 논문 또는 기술 문서 PDF

### 하이라이트 특성
- 단일 줄 하이라이트
- 여러 줄에 걸친 하이라이트
- 한 페이지에 하이라이트가 여러 개 있는 경우
- 여러 페이지에 하이라이트가 분산된 경우

### 텍스트 구조
- 본문 위주 문서
- 제목/소제목 구조가 뚜렷한 문서
- 표, 수식, 이미지가 포함된 문서

## 4. 우선 확보해야 할 정상 케이스

| ID | 유형 | 설명 | 우선순위 | 기대 결과 |
| --- | --- | --- | --- | --- |
| N1 | 교재 | 텍스트 레이어가 정상이고 단일 줄 하이라이트가 있는 문서 | 높음 | 하이라이트 좌표와 텍스트가 정확히 매핑된다 |
| N2 | 교재 | 여러 줄 하이라이트가 포함된 문서 | 높음 | 줄바꿈이 있더라도 하나의 의미 단위로 묶인다 |
| N3 | 슬라이드 | 페이지별 텍스트 블록이 짧은 문서 | 중간 | 제목/핵심 bullet 중심으로 구조화된다 |
| N4 | 논문 | 2단 컬럼 또는 복잡한 본문 구조를 가진 문서 | 중간 | 컬럼 구조에서도 잘못된 문장 결합이 줄어든다 |
| N5 | 혼합 | 여러 페이지에 하이라이트가 분산된 문서 | 높음 | 페이지 순서대로 하이라이트가 정렬된다 |

## 5. 우선 확보해야 할 예외 케이스

| ID | 유형 | 설명 | 우선순위 | 기대 처리 방식 |
| --- | --- | --- | --- | --- |
| E1 | 무하이라이트 | PDF에 하이라이트가 전혀 없음 | 높음 | "하이라이트가 없습니다" 메시지 반환 |
| E2 | 스캔본 | 이미지 기반 PDF라 텍스트 레이어가 없음 | 높음 | 지원 불가 또는 OCR 미지원 안내 |
| E3 | 손상 파일 | 업로드는 되지만 파싱이 실패하는 PDF | 높음 | 파일 파싱 실패 코드와 안내 메시지 반환 |
| E4 | 특수문자 | 수식, 기호, 특수문자가 많은 문서 | 중간 | 텍스트 정규화 규칙 확인 필요 |
| E5 | 비정상 annotation | 하이라이트 annotation은 있으나 텍스트 추출이 불안정한 문서 | 높음 | 부분 실패로 기록하고 원인 로그 남김 |

## 6. 샘플별 기록 항목

각 PDF는 아래 메타데이터를 함께 기록한다.

| 항목 | 설명 |
| --- | --- |
| sample_id | 샘플 식별자 |
| 파일명 | 원본 파일명 |
| 문서 유형 | 교재 / 슬라이드 / 논문 / 기타 |
| 페이지 수 | 전체 페이지 수 |
| 하이라이트 수 | 전체 하이라이트 개수 |
| 하이라이트 형태 | 단일 줄 / 다중 줄 / 다중 페이지 |
| 텍스트 레이어 | 있음 / 없음 / 불안정 |
| 표/수식 포함 여부 | 있음 / 없음 |
| 예상 난이도 | 낮음 / 중간 / 높음 |
| 기대 결과 | 정상 추출 / 부분 추출 / 실패 안내 |
| 비고 | 특이사항 |

## 7. 샘플 인벤토리 템플릿

아래 표를 실제 수집 현황 관리용으로 사용한다.

| sample_id | 파일명 | 문서 유형 | 하이라이트 수 | 텍스트 레이어 | 예상 난이도 | 기대 결과 | 상태 | 비고 |
| --- | --- | --- | --- | --- | --- | --- | --- | --- |
| SAMPLE-001 | highlighted.pdf | 합성 교재 | 1 | 있음 | 낮음 | 정상 추출 | 테스트 fixture 완료 | backend 테스트에서 생성 |
| SAMPLE-002 | blank.pdf | 합성 교재 | 0 | 있음 | 낮음 | 실패 안내 | 테스트 fixture 완료 | `NO_HIGHLIGHTS` 확인 |
| SAMPLE-003 | image-only.pdf | 합성 스캔본 | 0 | 없음 | 중간 | 지원 불가 | 테스트 fixture 완료 | `UNSUPPORTED_PDF` 확인 |
| SAMPLE-004 | broken.pdf | 손상 파일 | 0 | 불명 | 낮음 | 파싱 실패 | 테스트 fixture 완료 | `PARSING_FAILED` 확인 |
| SAMPLE-005 | lecture-notes.pdf | 실제 강의자료 예정 | 미정 | 미정 | 중간 | 정상 추출 | 수집 예정 | 외부 샘플 확보 필요 |

## 8. 테스트 우선순위

1. N1, N2, E1, E2를 먼저 확보한다.
2. annotation 추출 스파이크는 N1과 N2로 시작한다.
3. 파싱 실패/지원 불가 응답은 E1, E2, E3로 검증한다.
4. 노트 구조 생성 품질은 N3, N4, N5에서 확인한다.

## 9. 완료 기준

- 정상 케이스 2종 이상이 fixture 또는 실제 문서로 확보되어 있다.
- 예외 케이스 2종 이상이 fixture 또는 실제 문서로 확보되어 있다.
- 각 샘플에 대해 기대 결과가 표에 기록되어 있다.
- 스파이크 대상 우선순위가 정해져 있다.
