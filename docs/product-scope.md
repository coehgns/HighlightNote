# HighlightNote 제품 범위 문서

## 1. 문서 목적

이 문서는 HighlightNote MVP의 범위, 제외 범위, 핵심 사용자 흐름, 실패 처리 기준을 고정하기 위한 기준 문서다.
개발 중 기능 추가 요청이 생기더라도 이 문서에 없는 항목은 기본적으로 MVP 밖으로 본다.

## 2. 문제 정의

학생과 문서 기반 학습 사용자는 PDF에서 중요한 부분을 하이라이트하지만,
그 내용을 시험 대비용 노트나 요약 자료로 다시 정리하는 데 많은 시간을 쓴다.

기존 AI 요약 서비스는 문서 전체를 다시 읽고 중요도를 자체 판단해 요약하기 때문에,
사용자가 직접 강조한 정보와 결과물이 어긋날 수 있다.

HighlightNote는 "사용자가 표시한 하이라이트"를 중심으로 학습 노트 초안을 생성해
정리 시간을 줄이고 학습 의도를 더 잘 반영하는 것을 목표로 한다.

## 3. 목표 사용자

### 1차 타겟
- PDF 교재를 활용해 공부하는 대학생
- 시험 대비 요약 노트를 짧은 시간 안에 만들고 싶은 사용자

### 2차 타겟
- 논문이나 기술 문서를 정리하는 사용자
- 강의 슬라이드 핵심 내용을 재구성하려는 사용자

## 4. MVP 목표

MVP는 아래 한 흐름을 안정적으로 완성하는 것이 목표다.

1. 사용자가 하이라이트가 포함된 PDF를 업로드한다.
2. 시스템이 하이라이트 annotation과 텍스트를 추출한다.
3. 시스템이 하이라이트를 구조화해 노트 초안을 생성한다.
4. 사용자가 웹에서 결과를 확인한다.
5. 사용자가 PDF로 결과를 다운로드한다.

## 5. MVP 포함 범위

### 필수 기능
- PDF 업로드
- 하이라이트 annotation 추출
- 하이라이트 텍스트 및 위치 정보 매핑
- 하이라이트 기반 노트 구조 생성
- 노트 미리보기
- PDF 다운로드

### 사용자에게 보여야 하는 상태
- 업로드 전
- 업로드 중
- 분석 중
- 생성 완료
- 무하이라이트
- 지원 불가 PDF
- 파싱 실패

### 저장해야 하는 핵심 데이터
- 문서 메타데이터
- 하이라이트 원문 텍스트
- 하이라이트 페이지/좌표 정보
- 생성 상태
- 생성된 노트 섹션
- PDF export 이력

## 6. MVP 제외 범위

- Word 다운로드
- 사용자 계정 및 로그인
- 여러 사용자 협업
- 공유 링크
- OCR 기반 스캔본 지원
- 모바일 앱
- 전체 문서 일반 요약
- 템플릿 다변화

## 7. 제품 원칙

1. 하이라이트 우선
사용자가 표시한 영역이 가장 중요한 입력값이다.

2. 근거 노출
생성된 노트의 각 섹션은 어떤 하이라이트에서 왔는지 추적 가능해야 한다.

3. 실패 구분
무하이라이트, 스캔본, 파싱 실패는 서로 다른 상태로 다뤄야 한다.

4. 규칙 기반 우선
초기 버전은 규칙 기반 구조화만으로도 동작해야 하며, LLM은 선택적 후처리다.

5. 데모 가능성 우선
기능 수보다 완결된 사용자 경험이 더 중요하다.

## 8. 지원 문서 범위

### 지원 대상
- 텍스트 레이어가 있는 PDF
- annotation 기반 하이라이트가 포함된 PDF
- 교재, 슬라이드, 논문 형식의 일반 문서 PDF

### 비지원 대상
- 이미지 기반 스캔 PDF
- 하이라이트 없이 밑줄이나 자유형 드로잉만 있는 PDF
- 비정상 annotation 구조로 텍스트 매핑이 불가능한 PDF

## 9. 기능 요구사항

### FR-1. 업로드
- 사용자는 PDF 파일 1개를 업로드할 수 있다.
- 기본 업로드 제한은 20MB로 둔다.

### FR-2. 분석
- 시스템은 업로드된 PDF에서 annotation 하이라이트를 탐지한다.
- 시스템은 하이라이트 좌표와 텍스트를 함께 저장한다.

### FR-3. 노트 생성
- 시스템은 하이라이트를 문서 순서대로 정렬한다.
- 시스템은 하이라이트를 제목, 핵심 개념, 설명 단위로 구조화한다.

### FR-4. 결과 확인
- 사용자는 생성된 노트 초안을 웹에서 본다.
- 사용자는 각 노트 섹션의 원본 하이라이트 근거를 확인할 수 있다.

### FR-5. 내보내기
- 사용자는 생성된 노트를 PDF로 다운로드할 수 있다.

## 10. 비기능 요구사항

### NFR-1. 설명 가능한 실패
- 사용자에게 실패 사유를 상태별로 다르게 보여줘야 한다.

### NFR-2. 추적 가능성
- 개발자는 문서 ID 기준으로 분석 상태와 실패 원인을 확인할 수 있어야 한다.

### NFR-3. 데모 안정성
- 샘플 PDF 3종 이상에서 같은 흐름이 반복 재현돼야 한다.

### NFR-4. 확장 가능성
- 나중에 비동기 워커를 붙일 수 있도록 `note_jobs` 중심 상태 모델을 유지한다.

## 11. 실패 상태 정책

| 코드 | 의미 | 사용자 메시지 방향 |
| --- | --- | --- |
| `NO_HIGHLIGHTS` | 하이라이트가 없음 | 하이라이트가 포함된 PDF를 업로드해 달라고 안내 |
| `UNSUPPORTED_PDF` | 텍스트 레이어 없음 또는 비지원 구조 | 현재 지원 문서 형식을 안내 |
| `PARSING_FAILED` | 파싱 도중 오류 | 다시 시도 또는 다른 파일 업로드를 안내 |

## 12. MVP 완료 기준

- 샘플 PDF 3종 이상에서 업로드부터 PDF 다운로드까지 완료된다.
- 무하이라이트 PDF 1종 이상에서 실패 메시지가 분리돼 노출된다.
- 텍스트 레이어 없는 PDF 1종 이상에서 지원 불가 메시지가 분리돼 노출된다.
- 생성 결과에 원본 하이라이트 근거가 함께 보인다.

## 13. 출시 후 우선순위

2. 사용자 수동 편집 후 재다운로드
3. 하이라이트 색상 의미 반영
4. OCR 지원
5. 템플릿 다변화
