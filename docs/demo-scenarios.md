# HighlightNote 데모 시나리오

## 목적

이 문서는 MVP 발표와 수동 검증에 사용할 대표 시나리오를 고정한다.
데모는 복잡한 예외보다 "하이라이트 기반 노트 생성"의 핵심 가치를 빠르게 보여주는 방향으로 설계한다.

## 시나리오 1. 교재 하이라이트 기반 시험 대비 노트 생성

### 사용자
- 대학생

### 입력 문서 조건
- 텍스트 레이어가 있는 교재 PDF
- 단일 줄, 다중 줄 하이라이트가 함께 존재
- 최소 5개 이상 하이라이트 포함

### 사용자 스토리
- 사용자는 시험 범위 PDF에서 중요한 부분을 하이라이트했다.
- 사용자는 하이라이트를 기준으로 빠르게 정리 노트를 만들고 싶다.

### 기대 흐름
1. 사용자가 PDF를 업로드한다.
2. 시스템이 하이라이트를 추출한다.
3. 시스템이 하이라이트를 제목, 핵심 개념, 설명 구조로 묶는다.
4. 사용자가 웹에서 노트 초안을 확인한다.
5. 사용자가 PDF로 다운로드한다.

### 데모에서 강조할 포인트
- 원본 문장 전체가 아니라 사용자가 표시한 부분 중심으로 결과가 나온다.
- 결과 섹션마다 원본 하이라이트 근거가 연결돼 있다.

## 시나리오 2. 논문/기술 문서 핵심 정리

### 사용자
- 논문이나 기술 문서를 읽는 사용자

### 입력 문서 조건
- 텍스트 레이어가 있는 기술 문서 PDF
- 여러 페이지에 하이라이트가 분산
- 소제목 구조가 어느 정도 존재

### 사용자 스토리
- 사용자는 긴 기술 문서에서 핵심 문장을 하이라이트해 두었다.
- 사용자는 나중에 복습할 수 있도록 핵심 개념 중심 노트를 만들고 싶다.

### 기대 흐름
1. 사용자가 하이라이트된 기술 문서를 업로드한다.
2. 시스템이 페이지 순서대로 하이라이트를 정렬한다.
3. 시스템이 개념 단위로 노트 초안을 생성한다.
4. 사용자가 결과를 검토하고 PDF로 저장한다.

### 데모에서 강조할 포인트
- 여러 페이지에 흩어진 하이라이트도 문서 흐름대로 정리된다.
- 요약이 아니라 복습용 구조화에 가깝다는 점을 보여준다.

## 시나리오 3. 실패 처리 데모

### 목적
- 서비스가 실패를 숨기지 않고 설명한다는 점을 보여준다.

### 케이스 A. 하이라이트 없음
- 입력: 하이라이트가 없는 PDF
- 기대 결과: `NO_HIGHLIGHTS` 상태와 안내 메시지 노출

### 케이스 B. 텍스트 레이어 없음
- 입력: 스캔본 PDF
- 기대 결과: `UNSUPPORTED_PDF` 상태와 지원 불가 메시지 노출

## 데모 순서 권장안

1. 정상 케이스 1개로 핵심 가치를 먼저 보여준다.
2. 정상 케이스 2개로 문서 유형 확장성을 보여준다.
3. 마지막에 실패 케이스 1개를 보여주며 처리 정책을 설명한다.

## 데모 준비 체크리스트

- 샘플 PDF 3종 준비
- 업로드 시간과 생성 시간을 미리 점검
- 결과 PDF 저장 경로 확인
- 실패 케이스 메시지 문구 확정
